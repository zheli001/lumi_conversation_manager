## 1. Executive Summary

As Large Language Models (LLMs) proliferate in enterprise Java backends, two critical problems emerge:

1. **Context window limits** — LLM APIs impose strict Token limits (e.g. 128k for GPT-4o, 200k for Claude 3.5). Long conversations inevitably overflow, and naive truncation destroys coherence.
2. **Cost explosion** — Every token sent to the LLM API costs money. Without intelligent compression, multi-turn dialogues send redundant, repeated context on every call.

**MemoryFunnel** is a lightweight, zero-external-dependency Java local library that solves both problems through:

- A **Shadow-Buffer architecture** that separates active writes from background compression — guaranteeing near-zero write latency even during heavy context operations.
- A **Task-Aware Eviction Engine** that understands conversation lifecycle and discards completed-task memory intelligently, not blindly.
- A **SPI (Service Provider Interface) framework** that makes every behaviour — storage, tokenization, compression strategy, security — replaceable without touching core logic.

MemoryFunnel is **not** a general LLM SDK. It is the missing middleware layer between your Java application and any LLM API.

## 2. Problem Statement

### 2.1 The Context Window Crisis

LLM APIs accept a bounded context window measured in tokens. When a conversation exceeds this window, one of two failures occurs:

| Failure Mode | Consequence |
|---|---|
| Naive truncation (cut oldest) | Discards system prompt and initial instructions — LLM "forgets" its role |
| Hard API error (400) | Application crash; poor user experience |

Neither is acceptable for production systems.

### 2.2 Token Cost Waste

Without lifecycle-aware memory management, every LLM call sends:
- Completed intermediate reasoning steps (irrelevant after task closes)
- Redundant filler messages ("OK", "Got it", "Thank you")
- Duplicate system prompt repetitions

Industry estimates show **30–60% of tokens sent per call are semantically redundant** once a task completes.

### 2.3 Prompt Security Gap

Most libraries naively forward the entire conversation history to the LLM, including any PII (Personally Identifiable Information) the user may have disclosed (credit card numbers, medical data, credentials). There is no standard interception layer.

### 2.4 Java Ecosystem Gap

The Java enterprise ecosystem lacks a production-grade, framework-agnostic conversation state manager. Existing solutions (LangChain4j, Spring AI) are full-stack frameworks with heavy opinions. There is no equivalent of a focused, embeddable middleware library.

## 3. Human-AI Collaborative Development (HAC-Flow)

MemoryFunnel itself was designed using a novel development methodology — **HAC-Flow (Human-Agent Collaboration Flow)** — which this white paper documents as a reusable process.

### 3.1 HAC-Flow Overview

```
Input: Vague human intent
  ↓
[Stage 1] Intent Probing
  Ambiguity_Report.yaml generated by agents
  ↓
[Stage 2] Knowledge Alignment
  Multi-stakeholder clarification dialogue
  Frozen_Requirement.yaml produced
  ↓
[Stage 3] Parallel Design
  Architect / Security / SRE / PO agents debate concurrently
  Conflict_Resolution_Log.json generated
  ↓
[Stage 4] Parallel Implementation
  Dev Agent (code) + QA Agent (tests) run simultaneously
  ↓
[Stage 5] Human Arbitration
  Human reviews conflict reports and makes final decisions
  ↓
Output: Production-grade artefacts + ADR documentation
```

### 3.2 Core HAC-Flow Principles

| Principle | Description |
|---|---|
| **No Assumptions** | Agents must stop and clarify ambiguity — never auto-select a technical path without human confirmation |
| **Clarification First** | Before any design work, an `Ambiguity_Report` must be produced and resolved |
| **Conflict is Healthy** | Multiple agents with opposing viewpoints (cost vs quality, security vs performance) produce better outcomes than single-agent consensus |
| **Human as Arbitrator** | Humans decide on trade-offs; agents execute and present options |
| **Recursive Refinement** | Each design decision may trigger a new clarification cycle |

### 3.3 Key HAC-Flow Artefacts

| Artefact | Producer | Purpose |
|---|---|---|
| `Ambiguity_Report.yaml` | All Agents | Lists all unclear requirements before design begins |
| `Conflict_Ticket.yaml` | Conflicting Agents | Raised when agents detect a constraint violation |
| `Decision_Tree.md` | Human + Agents | Records all "why" decisions — more important than code |
| `Frozen_Requirement.yaml` | Human approval | Locked requirements after full clarification round |
| `Impact_Analysis.md` | Architect Agent | Generated when new requirements break existing architecture |

## 5. Product Scope & Requirements

### 5.1 In Scope

| Feature | Description |
|---|---|
| Context window management | Maintain conversation within Token limits without destroying coherence |
| Intelligent compression | Semantic summarisation triggered at Token threshold |
| Task-based eviction | Forget completed-task intermediate messages; preserve results |
| General eviction policies | Sliding window (by Token count), time-based, importance-score-based |
| Prompt security | Block PII/sensitive data from being sent to LLM unless explicitly opted in |
| Multimedia support | Metadata-first handling of images, files, audio references (no binary storage) |
| Session checkpoint | Export/restore full conversation state as a serialisable POJO |
| SPI extensibility | Pluggable storage, tokenizer, retention policy, sanitizer, encryptor |
| POJO-first data model | All data structures serialisable with Jackson/Gson |
| Java 17 reference implementation | Fully functional on Java 17 LTS |

### 5.2 Out of Scope

| Feature | Reason |
|---|---|
| Built-in LLM SDK (OpenAI, Anthropic) | Framework-agnostic; user integrates their own LLM client |
| Spring / Quarkus auto-configuration | Core library is pure Java; Spring Starter is a separate optional module |
| Vector database integration | Core library; vector search is an optional extension |
| Distributed session synchronisation | Phase 2; current scope is single-JVM |
| Arbitrary message position cut/rollback | Task eviction mechanism makes this unnecessary; adds too much complexity |

### 5.3 Confirmed Requirement Decisions

The following were clarified during the HAC-Flow sessions:

| Question | Decision | Rationale |
|---|---|---|
| Python or Java? | **Java 17 (→ 21)** | Enterprise Java ecosystem; no Python |
| Local library or remote service? | **Local library (.jar)** | Zero network dependency in core |
| Async model? | **CompletableFuture + ExecutorService abstraction** | Compatible with both thread pools (Java 17) and Virtual Threads (Java 21) |
| Storage format? | **JSON-compatible POJO** | Easy debugging, Jackson/Gson compatible |
| Multi-tenant session isolation? | **Session-ID + User-ID key namespacing via SPI** | Caller responsible for namespacing |
| Arbitrary rollback? | **Not supported** | Task eviction + sequence rollback covers the use case; arbitrary position cut is over-engineering |
| Double-buffer memory overhead? | **Accepted — by design** | Space-for-time trade-off; documented constraint |

## 6. Core Feature Specification

### 6.1 Three-Layer Memory Funnel Architecture

```
┌─────────────────────────────────────────────────────────────┐
│  L1: Active Buffer (Hot Zone)                               │
│  • Last N rounds of raw messages                            │
│  • All writes land here first                               │
│  • Never compressed — always current                        │
├─────────────────────────────────────────────────────────────┤
│  L2: Condensed Synopsis (Warm Zone)                         │
│  • Semantic summary of older history                        │
│  • Produced by Shadow-Buffer compression                    │
│  • Replaces evicted L1 messages                             │
├─────────────────────────────────────────────────────────────┤
│  L3: Deep Archival (Cold Zone) [Optional SPI]               │
│  • Encrypted, PII-isolated storage                          │
│  • Not included in LLM context by default                   │
│  • Accessible via explicit retrieval call                   │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 Shadow-Buffer Concurrency Model

- **Primary Buffer**: always writable; new messages always accepted
- **Shadow Buffer**: cloned asynchronously when Token threshold is reached; compression runs on shadow without blocking primary
- **Delta Patch Merge**: after compression completes, messages added to primary during compression are extracted as a delta and appended to the compressed result before atomic swap
- **Write Queue**: messages queue during the brief atomic merge window; queue overflow is the only condition causing full blocking
- **Rollback**: sequence-ID-anchored rollback cancels shadow task and truncates primary from target sequence onwards

### 6.3 Task-Aware Eviction

Every `ChatMessage` carries an optional `taskId`. When a task is marked complete (explicitly via API or detected by keyword signal):

1. All intermediate reasoning messages for that `taskId` are marked `obsolete`
2. On next `getContext()` call, obsolete messages are physically removed
3. Task result messages (final answers) are retained in L2 synopsis

### 6.4 Prompt Security (Core Requirement)

- A `Sanitizer` SPI intercepts every message **before** it enters the context sent to LLM
- Default policy: **block** — any message tagged `sensitive=true` is replaced with `[REDACTED]` in the outbound context
- User must explicitly configure a `PassthroughSanitizer` or custom `Sanitizer` to allow sensitive data
- `Sanitizer` runs on both individual messages and on L2 summaries before they are assembled for the LLM call

### 6.5 Multimedia Content (Metadata-First)

- Binary content is **never stored** in the conversation manager
- Images, files, audio are represented as `ContentBlock` with type, URI/reference, MIME type, and size
- On eviction, image `ContentBlock` references are removed; their accompanying text descriptions are preserved
- Checkpoint snapshots include `ContentBlock` references; a `Path Validator` confirms references are still valid on restore

### 6.6 Session Checkpoint

- `SessionSnapshot` captures: current primary buffer, task state map, last sequence ID, engine configuration
- Exported as POJO → serialisable to JSON, Protobuf, or any format via user's chosen serialiser
- Each snapshot includes a content hash checksum; corrupt or tampered snapshots are rejected on load
- Checkpoint is triggered: explicitly via `manager.checkpoint()`, on session interruption, or on task boundary

## 7. Non-Functional Requirements & Extension Points

These are **not** hardcoded into the core engine. They are pluggable via SPI:

| Concern | SPI | Default Behaviour |
|---|---|---|
| Thread safety | Core `AtomicReference` + `StampedLock` (built-in) | Always on; non-negotiable in core engine |
| Storage | `ChatStorage` SPI | `InMemoryStorage` |
| Tokenization | `TokenCounter` SPI | `CharacterBasedTokenCounter` (1 char ≈ 0.25 tokens estimate) |
| Retention / Eviction | `RetentionPolicy` SPI | Sliding window by Token count |
| Compression / Summarisation | `Summarizer` SPI | No-op (must be provided by user for semantic compression) |
| PII sanitisation | `Sanitizer` SPI | Block-all (replace sensitive content with `[REDACTED]`) |
| Encryption (checkpoint) | `Encryptor` SPI | No encryption (plaintext; enterprise deployment should inject `AESEncryptor`) |
| Metrics / Observability | `MetricsProvider` SPI | No-op |
| Concurrency executor | `ExecutorFactory` SPI | `Executors.newCachedThreadPool()` → swap for Virtual Threads in Java 21 |

## 8. Technology Constraints & Migration Path

### 8.1 Current: Java 17

- Concurrency: `AtomicReference`, `StampedLock`, `CompletableFuture`, `ExecutorService`
- No Virtual Threads (Project Loom); thread pool via `ExecutorFactory` SPI
- All concurrent primitives use standard `java.util.concurrent`

### 8.2 Near Future: Java 21+

The `ExecutorFactory` SPI is the single migration point:

```java
// Java 17 (current)
ConversationManager.builder()
    .withExecutor(Executors.newCachedThreadPool())
    .build();

// Java 21 (zero code change in business logic)
ConversationManager.builder()
    .withExecutor(Executors.newVirtualThreadPerTaskExecutor())
    .build();
```

No other code changes required. The `Shadow-Buffer` async tasks are submitted to the injected executor; the switch to Virtual Threads is fully transparent.

### 8.3 Future: Kotlin Flow / Reactive Streams

The core engine operates on `ChatMessage` POJOs and returns `CompletableFuture<T>`. An adapter layer (not in core) will wrap these for:

- Kotlin `Flow` (via `kotlinx.coroutines` bridge)
- Project Reactor `Mono`/`Flux` (via `reactor-adapter`)

The core API is stable; adapters extend outward.

## 9. HAC-Flow vs Traditional Development

| Dimension | Traditional SDLC | HAC-Flow |
|---|---|---|
| **Requirements** | Documents written upfront; misunderstandings surface in testing | Recursive clarification with agent-driven ambiguity scanning; gaps found before design |
| **Security** | Security audit as a final gate; findings require expensive rework | Security agent participates from day 1; PII handling baked into SPI contract |
| **Performance** | SRE consulted post-implementation when OOM occurs | SRE agent defines memory thresholds before first line of code |
| **Cost awareness** | CFO sees API costs after deployment | Cost agent challenges token-expensive decisions in architecture phase |
| **Conflict resolution** | Conflicts surfaced in code review or production | Conflicts resolved in design phase via structured `Conflict_Ticket.yaml` |
| **Documentation** | Written after the fact; often stale | Decision_Tree.md and ADRs generated during design; always current |
| **Delivery** | Estimated 4–6 weeks for this scope | Estimated 11 working days (design + implementation) |
| **Rework rate** | High — architecture changes after coding starts are expensive | Low — architecture locked before implementation; changes are additive |

### 9.1 Quantified Impact on MemoryFunnel

| Issue | How It Was Caught | When (HAC) | vs Traditional |
|---|---|---|---|
| AsyncIO/blocking model mismatch | Architect vs SRE debate | Design phase | Integration testing |
| PII data in LLM context | Security agent | API design phase | Post-breach audit |
| Token overhead from summarisation | CFO challenge | Architecture phase | After first billing cycle |
| OOM from double-buffer | SRE raised memory math | Design phase | Production OOM |
| Streaming output state corruption | QA agent | API design phase | Production data corruption |

## 10. Stakeholder Conflict Resolution Log

Key conflicts resolved during HAC-Flow sessions:

| Conflict | Parties | Decision |
|---|---|---|
| Local tokenizer (adds 1-2MB) vs remote API call | Architect vs SRE | **Local tokenizer** — latency and offline availability trump size |
| Auto-decision by agent vs mandatory human approval | Architect vs User | **No auto-decisions on architecture** — agents always surface options for human arbitration |
| Optimistic write (chunks to Redis during stream) vs double-buffering | Architect vs QA | **Double-buffering** — dirty reads cause LLM hallucinations; consistency over performance |
| Physical memory zero-fill vs soft-delete | Security vs SRE/CFO | **Tiered** — standard mode: null reference; enterprise mode: `Arrays.fill` zero-fill (cost-disclosed) |
| Arbitrary rollback support | Architect vs simplicity | **Not supported** — task eviction achieves the same semantic result with far less complexity |
| Memory overhead of shadow buffer | SRE vs Architect | **Accepted by design** — space-for-time; documented; configurable disable with graceful degradation |
| Kotlin Flow / reactive compatibility | PO vs Architect | **Adapter layer** — core API remains POJO + CompletableFuture; reactive wrappers are optional extensions |
